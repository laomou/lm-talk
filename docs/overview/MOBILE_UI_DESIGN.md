# LM Talk 移动端 / Web UI 设计备忘

本文记录当前已确认的产品 UI 方向，方便后续实现时保持一致。这里先写设计，不等于已经实现。

> 当前范围调整：**先不做群聊**。群聊、群邀请、创建群聊、群聊列表、群聊会话都放到后续迭代。

## 1. 设计原则

- **产品优先**：首页只展示用户能理解、能直接操作的内容。
- **三页主结构**：主导航只保留 `聊天 / 通讯录 / 我`。
- **先做单聊**：当前 MVP 只做注册、登录、导入、加好友、单聊、文件/Emoji、个人与安全设置。
- **群聊后置**：群聊相关入口先隐藏或不实现，避免半成品干扰主流程。
- **附件先轻量**：MVP 两端附件面板只做 `图片` + `文件`，拍摄、相册、位置、其他等六宫格能力留待后续迭代。
- **技术细节后置**：Mailbox、DHT、PreKey、同步诊断等实现细节不出现在聊天和通讯录首页；统一收纳到 `我 -> 同步与安全` 或更深层页面。
- **信任结论前置**：对 E2EE 产品来说，`对方指纹是否已核验` 是用户必须知道的信任结论，不是技术噪音；会话头部和联系人行保留极简信任标识，点击后再进入核验详情。
- **消息状态可见**：异步 Mailbox / 网络投递下，用户必须知道消息是否发出、送达、已读或失败；气泡保留极简投递状态。
- **信息分工清晰**：
  - `聊天`：最近单聊会话和消息收发。
  - `通讯录`：好友、好友申请、添加好友、联系人详情。
  - `我`：个人资料、身份、安全、同步、设置、关于。
- **移动端优先**：先保证手机上的导航、层级、触控和消息输入体验，再做 Web 适配。

规则：

- 聊天列表当前只展示单聊最近会话。
- 右侧不放添加按钮；添加好友属于通讯录。
- 会话行第一行：左侧昵称，昵称旁显示极简信任标识，右侧时间，时间和昵称在同一行对齐。

- 昵称太长使用省略号。
- 最新消息太长使用省略号。
- 空状态只提示“暂无聊天”，不要展示“第一次使用建议”三步引导。

### 4.2 单聊页

进入具体聊天后隐藏底部 Tab，页面变为全屏会话。

```text
┌────────────────────────────┐
│ ‹       张三很长... ✓    ⋯ │
├────────────────────────────┤
│                            │
│  对方消息气泡               │
│                            │
│            我的消息气泡  已送达│
│                            │
│  对方消息气泡               │
│                            │
├────────────────────────────┤
│ ＋  [ 输入消息…      ]  😊 ↑ │
└────────────────────────────┘
```

头部规则：

- 左侧返回。
- 中间显示联系人名，过长省略。
- 联系人名旁保留极简信任标识：`✓` 表示已核验，`⚠️` 表示未核验。
- 点击信任标识或联系人名区域，进入指纹核验 / 安全详情。
- 右侧 `⋯` 进入聊天设置或更多操作。
- 不在聊天头部展示同步状态、DHT 状态、PreKey 状态等实现细节。

消息气泡状态：

| 状态 | 展示 | 含义 | 用户操作 |
| --- | --- | --- | --- |
| 已发送 | `已发送` 或单勾 | 本机已加密并提交发送队列 / 已交给投递层 | 一般无需操作 |
| 已送达 | `已送达` 或双勾 | 对方设备或 Mailbox delivery ACK 已确认收到 | 一般无需操作 |
| 已读 | `已读` 或高亮双勾 | 对方已打开或发送 read receipt | 一般无需操作 |
| 失败 | `失败` / `!` | 加密、网络、Mailbox 或信任检查导致未成功投递 | 可点击查看原因并重试 |

规则：

- 消息状态是基本可用性，不是技术噪音。
- 自己发送的消息气泡右下角展示状态。
- 对方发来的消息一般不展示发送状态。
- `失败` 必须明显可见，并提供重试入口。
- 具体失败原因可以点开后再展示，例如网络失败、对方设备不可用、未核验无法发送等。
- 不在气泡旁直接展示 Mailbox ID、DHT 路由、PreKey 原始状态等实现细节。

输入区规则：

```text
＋  [输入消息…]  😊  ↑
```

- `＋`：附件入口，支持图片、文件等。
- `😊`：Emoji 入口。
- `↑`：发送按钮，使用圆形向上箭头图标。
- 输入为空时，发送按钮置灰或不可点击。
- 发送按钮不使用文字“发送”。

## 7. Web UI / 桌面端

Web 端沿用同一套产品分工：`聊天 / 通讯录 / 我`。当前目标是**两列布局**，不是三列。

### 7.1 Web 总原则

- Web 端当前目标：左侧 72px 窄图标 rail + 右侧主内容区。
- 通用页面不默认做三列，不做 `主导航 / 列表 / 详情` 常驻三栏。
- **桌面聊天例外**：聊天页采纳并列布局，保留现有 `.chat-shell` 双列：会话列表 + 当前会话并排显示。
- Web 端也不展示用户不理解的“收件箱”。
- 技术诊断入口仍然放在 `我 -> 同步与安全`。
- 当前不做群聊相关 Web 入口。
- 版本号不放侧栏，只放 `我 -> 关于`。

### 7.2 Web Shell

```text
┌──────┬────────────────────────────────────┐
│  LM  │ 当前页面标题                  操作区 │
├──────┼────────────────────────────────────┤
│  💬  │                                    │
│  👥  │             主内容区               │
│  🙂  │                                    │
│              │                                    │
│              │                                    │
└──────┴────────────────────────────────────┘
```

规则：

- 左侧保留 72px 窄图标 rail，不做宽品牌栏。
- rail 只放图标化主导航：`聊天 / 通讯录 / 我`；品牌可以用小图标/缩写，不占宽栏。
- 左侧不要常驻 `Version ...`、版本号、commit、诊断信息。
- 左侧不要加入诊断、收件箱、DHT、Mailbox、群聊等主导航项。
- 当前选中项高亮。
- 版本信息只放到 `我 -> 关于`。

```text

│              │ 头像 Alice ⚠️ 周一│          我的消息 已送达│
│              │   好的，明天见    │                      │

│              │                  │ ＋ [ 输入消息… ] 😊 ↑ │
```

规则：

- 桌面聊天不采用“单区切换”。
- `.chat-shell` 内部保留双列：`会话列表 / 当前会话`。
- 左侧 App 主导航仍然只放：`聊天 / 通讯录 / 我`。
- 会话列表规则和移动端一致：昵称、信任标识、时间同一行，长文本省略。
- 未选中会话时，右侧显示空状态：`选择一个聊天开始`。
- 点击会话后，右侧当前会话区域更新，不离开聊天页整体框架。
- 聊天区规则和移动端一致：`＋ / 输入框 / 😊 / ↑`。
- 附件和 Emoji 面板仍然在输入栏下方展开，占用真实高度，消息区收缩。
- Web MVP 附件面板同样只做 `图片` + `文件`；不做拍摄、位置、其他。
- Web 端不在聊天页放添加好友入口。

响应式规则：

- 桌面宽屏：会话列表和当前会话并列。
- 窄屏 / 手机：退化为移动端单区页面栈，点击会话后进入单聊页。

### 7.11 响应式断点建议

```text
>= 1024px：通用页面为 72px 窄图标 rail + 右侧主内容区；聊天页为 rail + `.chat-shell` 双列并列
768px - 1023px：平板两列或左侧图标栏 + 主内容区
< 768px：移动端单栏 + 底部 Tab
```

具体规则：

- 桌面：左侧 72px 窄图标 rail 常驻。
- 平板：优先保留窄图标 rail；空间不足时按移动端底部 Tab。
- 手机：使用底部 Tab；二级页面隐藏底部 Tab。
- 信息架构不因断点改变，只改变展示方式。

## 8. 内容归属规则

| 内容 | 当前放在哪里 | 当前不放在哪里 |
| --- | --- | --- |
| 最近单聊消息 | 聊天列表 | 通讯录首页 |
| 单聊收发 | 单聊会话页 | 通讯录首页 |
| 好友申请 | 通讯录 -> 新的朋友 | 收件箱、聊天页 |
| 好友列表 | 通讯录首页 A-Z | 聊天页 |
| 添加好友（粘贴名片） | 通讯录右上角添加页 | 聊天页 |
| 扫码添加 | 移动端添加页 | 桌面 Web |
| Mailbox / DHT / PreKey / 诊断 | 我 -> 同步与安全 | 通讯录、聊天首页 |
| 指纹核验结论 | 会话头部、联系人行、联系人详情 | 完全藏到二级/三级页面 |
| 消息投递状态 | 自己发送的消息气泡 | 完全隐藏或只放诊断页 |
| 版本号 | 我 -> 关于 | 侧栏、主 footer、聊天页、通讯录页 |
| 群聊 / 群邀请 / 创建群聊 | 后续迭代 | 当前 MVP 主流程 |

## 9. MVP 范围

优先实现：

- 三个主导航：聊天、通讯录、我。
- 聊天列表移动端结构。
- 单聊页基础输入区：`＋ / 输入框 / 😊 / ↑`。
- 附件与 Emoji 面板在输入栏下方展开。
- 附件 MVP 只做两项：图片、文件。
- 通讯录首页：新的朋友、好友列表 A-Z。
- 新的朋友页：好友申请。
- 联系人搜索页。
- 添加页：移动端两条入口 `添加好友（粘贴名片）` + `扫码添加`；桌面 Web 只显示 `添加好友（粘贴名片）`。
- 联系人详情页主操作 `发消息`。
- 会话头部、联系人行、联系人详情展示极简信任标识。
- 自己发送的消息气泡展示四态：已发送、已送达、已读、失败。
- 我页面基础结构。
- Web 使用 72px 窄图标 rail，不做宽品牌栏。
- Web 通用页面为窄 rail + 主内容区；桌面聊天保留 `.chat-shell` 会话列表 + 当前会话并列。

后续迭代：

- 群聊列表。
- 群聊会话。
- 创建群聊。
- 群邀请。
- 群资料和群成员管理。
- 附件六宫格：拍摄、相册、位置、其他等扩展入口。
- 右侧 A-Z 快速索引拖动定位。
- 置顶聊天。
- 会话免打扰、删除、标记未读。
- 聊天长按菜单。
- 搜索历史。
- 隐私模式。
- 更完整的同步与安全诊断页。

## 10. 当前不要做的事

- 不要做群聊入口。
- 不要做群聊会话。
- 不要做创建群聊。
- 不要做群邀请。
- 不要在通讯录首页展示收件箱。
- 不要把好友申请拆成用户不理解的技术投递概念。
- 不要在聊天列表右上角放添加按钮。
- 不要在主页面暴露 DHT、Mailbox、PreKey、sealed slot 等术语。
- 不要把“对方指纹是否已核验”完全藏到三级页面；它必须作为极简信任结论可见。
- 不要隐藏消息投递状态；至少要让用户知道已发送、已送达、已读或失败。
- 不要恢复“第一次使用建议先完成下面 3 步”的空状态。
- 不要把 Emoji 或附件面板覆盖在消息内容上。
- 不要在 MVP 做附件六宫格；先只做图片和文件。
- 不要把通讯录等通用页面默认做 Web 三列布局。
- 不要把桌面聊天改成单区切换；桌面聊天采用 `.chat-shell` 双列并列。
- 不要在 Web 侧栏常驻版本号。
- 不要做 Web 宽品牌侧栏；保留 72px 窄图标 rail。
- 不要在桌面 Web 展示扫码添加入口。
- 不要把 `粘贴名片` 做成独立第三入口；并入 `添加好友`。
