# LM Talk three-node federation template

This template runs three local `lm_node` instances behind Caddy reverse proxies so operators can test Mailbox + DHT snapshot sync and replication behavior before deploying public nodes.

## Ports

- node A: `http://localhost:8081`
- node B: `http://localhost:8082`
- node C: `http://localhost:8083`

Each node syncs with the other two through `sync_peers`.

## Quick start

```bash
cd deploy/lm-node-federation
mkdir -p secrets
for n in a b c; do
  openssl rand -base64 32 > "secrets/node-$n-token"
  openssl rand -base64 32 > "secrets/state-file-passphrase-$n"
  openssl rand -base64 32 > "secrets/state-db-passphrase-$n"
done
chmod 600 secrets/*
docker compose up -d --build
./smoke-test.sh
# Or run the full validation suite:
./run-all.sh
```

Use a node URL plus the matching token from `secrets/node-*-token` in the Web app sync settings.

## Public deployment notes

For real public nodes:

1. Replace the `Caddyfile.*` `:80` sites with real HTTPS hostnames.
2. Update each `sync_peers[].url` to the public HTTPS URL of the peer.
3. Use unique peer IDs, unique tokens, and encrypted persistent volumes.
4. Use `state_db_encryption_mode = external` with encrypted volumes, or build with `LM_NODE_CARGO_FEATURES=sqlcipher` and switch each node config to `state_db_encryption_mode = sqlcipher` for database-level encryption testing.
5. Set `cors_allow_origins` to your deployed Web origins.

This template is for federation/bootstrap testing. Run long-lived nodes with monitoring on `/health`, `/control/stats`, and `/control/metrics`.


## Smoke test

`./smoke-test.sh` checks that all three reverse-proxied nodes respond to `/health` and `/control/stats`, generates a signed smoke identity and ContactCard inside the node container, publishes and finds that ContactCard through node A DHT, pushes a signed Mailbox message to node A, exports a snapshot from node A, imports it into node B, verifies the ContactCard DHT record is findable from node B, takes the Mailbox message from node B, and checks node B sync status.

The script publishes a real signed ContactCard generated by `lm_node identity` / `lm_node contact-card`, so it also exercises server-side ContactCard DHT validation.


## Chaos smoke

`./chaos-smoke.sh` simulates a node-B outage: it stops node B, pushes five signed Mailbox messages to node A, restarts node B, imports node A's snapshot into node B, and verifies node B can take the recovered mailbox messages. This is a short functional chaos check, not a substitute for long-running load/partition testing.


## Load smoke

`./load-smoke.sh` pushes a short burst of signed Mailbox messages to node A, imports node A's snapshot into node C, verifies node C mailbox status, takes the messages from node C, and checks metrics are exposed. Set `MESSAGE_COUNT=100` (or another value) to increase the short burst size.


## Run all validation

`./run-all.sh` starts the compose stack unless `LM_NODE_FEDERATION_SKIP_UP=1` is set, writes a machine-readable JSON report to `federation-report.json` (or `LM_NODE_FEDERATION_REPORT`), then runs:

1. `smoke-test.sh`
2. `chaos-smoke.sh`
3. `load-smoke.sh`

Example:

```bash
./run-all.sh
MESSAGE_COUNT=100 ./run-all.sh
LM_NODE_FEDERATION_SKIP_UP=1 ./run-all.sh
LM_NODE_FEDERATION_REPORT=/tmp/lm-federation-report.json ./run-all.sh
```


## GitHub Actions

`.github/workflows/federation-smoke.yml` provides a manual `workflow_dispatch` job that creates fresh node secrets, runs `./run-all.sh`, uploads `federation-report.json` as an artifact, and dumps compose logs on failure. Use it for heavier federation validation outside the normal quick CI path.
